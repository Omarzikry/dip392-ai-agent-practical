TEST SCENARIOS (summary)

1. Happy-path: valid topic query → non-empty structured answer, matching topics, next steps.
2. Stats path: query with "mean"/"std" keyword → CalculatorTool called, numeric output shown.
3. Invalid input: empty, whitespace-only, too short (<3 chars), too long (>500 chars) → rejected.
4. No match: gibberish query → is_valid=True but topics_found=[], helpful fallback message shown.
5. Data flow: ValidatorTool output feeds SearchTool; FileReaderTool returns dict[str,str];
   SearchResult scores are positive floats; CalculatorTool returns typed StatsResult.
6. CLI exit codes: valid query exits 0; empty/invalid query exits 1; unknown topic exits 0.
7. CLI output: --verbose adds "Topics matched" line; stats query prints "Mean" in stdout.
8. Error handling: FileNotFoundError for missing KB file; ValueError for div-by-zero and empty list.

DEPLOYMENT PREPARATION

Install: pip install -r requirements.txt && pip install -e .
Run CLI: python3 -m study_agent.main "your topic here"
Run tests: python3 -m pytest tests/ -v
No external APIs or environment variables required.
GitHub Actions CI (.github/workflows/ci.yml) runs the full test suite on every push (Python 3.9 + 3.11).

DATA CONVERSION

Raw string → ValidatorTool (strip, validate) → keyword list (stop-word filter) →
FileReaderTool (Markdown → dict[str, str]) → SearchTool (keyword scoring → SearchResult list) →
AgentResponse dataclass → format_response() (CLI text) or JSON (FastAPI).

Typed Python dataclasses act as contracts between all components, ensuring format consistency
throughout the pipeline. The same AgentResponse object is used for both CLI and web output.
